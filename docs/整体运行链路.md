# 整体运行链路

## 一次真实请求怎么流动

把 TokenPilot 当成一个“夹在 OpenClaw 和上游模型之间的上下文控制层”会更容易理解。

主链路如下：

1. OpenClaw 加载 `tokenpilot` 插件
2. 插件注册 hook、provider、recovery tool
3. 用户请求进入本地 embedded proxy
4. 请求在发给上游模型前，先做 stable prefix 和 before-call reduction
5. 上游模型返回结果
6. transcript 被同步成 raw semantic turn
7. history 层更新任务注册表
8. decision 层判断任务状态、eviction、后续策略

## 对应代码位置

### 1. 插件入口

- `packages/openclaw-plugin/src/index.ts`

这是 composition root。当前它负责把很多模块组装起来。

### 2. runtime 注册

- `packages/openclaw-plugin/src/context-stack/integration/runtime-register.ts`

这里做的事：

- 注册 `memory_fault_recover` 工具
- 初始化 session topology
- 启动 embedded proxy
- 接入 OpenClaw 生命周期

### 3. embedded proxy

- `packages/openclaw-plugin/src/context-stack/integration/proxy-runtime.ts`

这里是最重要的请求入口之一。它负责：

- 解析 payload
- 解析或推断 session id
- 执行 stable prefix rewrite
- 注入 procedural memory
- 执行 before-call reduction
- 转发给上游模型
- 写 trace

### 4. request preprocessing

- `packages/openclaw-plugin/src/context-stack/request-preprocessing/`

里面的主要模块：

- `stable-prefix.ts`
- `before-call-reduction.ts`
- `after-call-reduction.ts`
- `tool-results-persist.ts`

### 5. page out

- `packages/openclaw-plugin/src/context-stack/page-out/`

这部分负责把 transcript 变成可持续维护的状态。

核心点：

- 同步 transcript
- 构建 raw semantic turn
- 重写 canonical history
- 触发 eviction

### 6. history 层

- `packages/layers/history/src/raw-semantic.ts`
- `packages/layers/history/src/registry.ts`

这层是真正持久化状态的地方。

### 7. decision 层

- `packages/layers/decision/src/policy.ts`
- `packages/layers/decision/src/task-state-estimator.ts`

这层负责决定：

- 哪些内容需要 reduction
- 哪些任务是 active / completed / evictable
- 何时推进 eviction

## 用 Python 思维理解

你可以把整条链路想成：

1. `openclaw-plugin` 是 Flask/FastAPI 的请求入口和中间件
2. `runtime-core` 是纯函数和公共算法库
3. `history` 是状态存储层
4. `decision` 是策略层

如果只抓主干，不要一开始陷进所有细节，先记住：

- `plugin` 负责接线
- `runtime-core` 负责执行
- `history` 负责记账
- `decision` 负责判断
