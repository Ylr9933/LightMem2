# 仓库总览

## 这是什么

TokenPilot 是一个挂在 OpenClaw 前后的上下文运行时。它的目标不是替代模型，而是控制长会话中的上下文：

- 在请求发出前，把明显冗余的内容变短
- 在会话持续进行时，维护任务状态
- 在任务完成后，把冷数据从主上下文里换出去
- 在必要时，允许通过恢复工具取回被裁剪或归档的内容

## 代码分层

### 1. `packages/kernel`

最底层的共享契约层。

主要放：

- 运行时模块接口
- 事件类型
- 上下文 segment 结构

可以把它理解成 Python 项目里的：

- `typing`
- `Protocol`
- 公共 dataclass / TypedDict

### 2. `packages/runtime-core`

通用运行能力层，不直接依赖 OpenClaw。

主要放：

- reduction pipeline
- 各类 reduction pass
- archive recovery
- tool result persist

它更像一个“公共算法与执行库”。

### 3. `packages/layers/history`

长期状态层。

主要放：

- raw semantic turn 的落盘
- session task registry
- canonical history rewrite / eviction
- delta view 构造

这一层相当于系统的状态仓库。

### 4. `packages/layers/decision`

策略与判断层。

主要放：

- policy 主模块
- reduction / eviction 分析器
- task-state estimator

这一层相当于系统的大脑。

### 5. `packages/openclaw-plugin`

OpenClaw 适配层，也是当前真实运行入口。

主要放：

- OpenClaw hook 注册
- 本地 responses proxy
- request preprocessing
- transcript sync
- recovery tool 注册

这一层负责把 OpenClaw 的请求翻译成 TokenPilot 的内部流程。

### 6. `experiments`

实验与 benchmark 层，不是主运行时。

主要放：

- pinchbench 脚本
- claw-eval 相关内容

## 当前项目状态

现在这个仓库已经不是“纯实验草稿”，但也还没有完全产品化。

当前比较成熟的是：

- 基本分层
- OpenClaw 插件接线
- reduction / recovery / task-state 主链路

当前还需要继续理顺的是：

- 插件入口文件仍然偏大
- benchmark 脚本与真实运行配置仍有耦合
- 文档和代码命名已经经历多次迁移，历史包袱较多
